# Memory Protocol (always loaded)

Project-scoped persistent memory lives in `.claude/memory/`, tracked in Git.
Source of truth for decisions, patterns, feedback, frictions, session journals.

## Structure

```
.claude/memory/
├── MEMORY.md             live digest, ~2 KB, loaded at session start
│                         auto-generated alongside INDEX.md
├── INDEX.md              full catalogue, ~6+ KB, grep-only (not auto-loaded)
├── TAGS.md               canonical tag vocabulary (closed set)
├── README.md             usage guide
├── decisions/            one file per architectural decision (ADR-style)
│                         release entries auto-segregated in INDEX
├── feedback/             owner feedback entries
├── patterns/             validated reusable patterns
├── frictions/            friction + resolution entries
└── sessions/             end-of-session journals
    └── archive/          older entries (moved via `pnpm memory:archive`)
```

## Two-tier catalogue (MEMORY.md + INDEX.md, since v6.1.0)

- **`MEMORY.md`** — live digest loaded by the SessionStart hook. Contains
  top 3 recent sessions + top 5 active non-release decisions + open
  frictions + active patterns + pointer to INDEX. ~2 KB target.
- **`INDEX.md`** — full catalogue with release log segregated in a
  collapsible section. Grep-only, not auto-loaded.

Both are regenerated by `pnpm memory:index` from the frontmatter of every
entry. `pnpm memory:index --check` fails CI if either drifts.

## Archival (optional, manual)

`pnpm memory:archive` relocates session journals older than 30 days into
`sessions/archive/`. Flags : `--days N` (custom threshold), `--dry-run`
(preview). The archive folder is skipped by the INDEX walker, so the
active list stays lean without losing history.

## When to READ

**Before any non-trivial task**, grep the relevant domain tag to surface prior context:
```bash
grep -rl "#<domain>" .claude/memory/
```
Read the top 3-5 matches before acting. Don't re-derive what's already decided.

**Always read on session start**:
- `.claude/memory/INDEX.md` (sommaire)
- Most recent file in `.claude/memory/sessions/`

## When to WRITE

| Event | Action |
|---|---|
| Architectural decision made | New file in `decisions/` immediately |
| Owner gives feedback (correction or validation) | New file in `feedback/` |
| Pattern validated as reusable | New file in `patterns/` |
| Friction encountered + resolved | New file in `frictions/` |
| End of session | Journal entry in `sessions/YYYY-MM-DD-HHMM.md` |
| Release cut | Auto-generated via `scripts/memory-record-release.js` |

### Trigger phrases — when to write a `feedback/` entry

Historical audit (2026-04-19) : **0 feedback entries en 30 jours d'activité**
alors que l'owner corrige régulièrement. La discipline ne tient pas quand
le déclencheur est flou. Formalisation :

**Correction** — l'owner utilise une de ces tournures ou équivalent :
- *"non, fais plutôt X"* · *"pas comme ça"* · *"arrête de faire X"*
- *"je t'ai déjà dit de..."* · *"je veux que tu..."*
- *"ce n'est pas ce que j'avais demandé"*
- *"tu n'écoutes pas"* · *"tu es complaisant"*
→ **Écrire immédiatement** `feedback/YYYY-MM-DD-<slug>.md` avec la règle,
la raison donnée par l'owner, et comment appliquer à l'avenir.

**Validation non-triviale** — l'owner approuve un choix **inhabituel** :
- Approuve une approche qu'il aurait pu contester (ex : "oui le bundle PR
  unique était le bon call")
- Rétrofeedback positif sur un pattern récent ("continue comme ça")
→ **Écrire** `feedback/YYYY-MM-DD-<slug>.md` — sinon tu risques de
dériver de ce qui a déjà marché.

**Ne pas écrire** pour les "ok merci" ou "bien" génériques — seulement
quand la validation porte sur un choix non-obvie.

### Trigger phrases — when to write a `patterns/` entry

**Pattern validé** si :
- Tu as codé un truc qui sera **re-utilisé tel quel** dans 2+ futurs cas
- Un pattern a été **explicitement demandé** par l'owner ("on fera comme
  ça désormais pour tous les X")
- Un refactor a produit une structure **plus claire** que la précédente
→ **Écrire** `patterns/YYYY-MM-DD-<slug>.md` avec : problème résolu,
template/code extrait, quand l'appliquer, anti-patterns à éviter.

### Trigger phrases — when to write a `frictions/` entry

**Friction** si tu as perdu plus de 5 min à :
- Contourner un outil / une API / une config
- Découvrir un piège non-documenté
- Débugger un truc que la rule prescrit mais qui ne marche pas
→ **Écrire** `frictions/YYYY-MM-DD-<slug>.md` avec : ce qui bloque, la
résolution trouvée, ce qu'il faut pour que ça ne se reproduise pas (rule
update ? script ? test ?).

### Session-wrap audit (mandatory)

Avant d'écrire le journal de session, réponds à ces 3 questions. Si la
réponse est OUI à l'une, écris l'entrée **avant** le journal :

1. **L'owner m'a-t-il corrigé ou validé un choix non-obvie ?** → `feedback/`
2. **Ai-je codé un pattern qui sera re-utilisé ?** → `patterns/`
3. **Ai-je perdu > 5 min sur un obstacle contournable ?** → `frictions/`

**Mandatory at end of session**: write journal entry summarizing actions, decisions, frictions, next steps. Run `pnpm memory:index` before session end.

## Entry format (YAML frontmatter required)

```yaml
---
id: <slug-unique-kebab-case>
date: YYYY-MM-DD
type: decision | pattern | friction | feedback | session
tags: [#domain-tag, #type-tag, #scope-tag, #priority-tag]
scope: template | client-specific
status: active | deprecated | superseded
supersedes: <id>   # optional, if replacing prior entry
---

# Title (short, imperative)

Body in markdown. Keep focused — one decision or observation per file.
Cross-link related entries: [other entry](../decisions/other.md).
```

## Tag vocabulary (strict closed set)

Defined in `.claude/memory/TAGS.md`. Using an unlisted tag = protocol violation.
If new domain needed: add to TAGS.md via a `decisions/` entry explaining why.

## Search patterns

```bash
# all design-related entries
grep -rl "#design" .claude/memory/

# active decisions only
grep -rl "status: active" .claude/memory/decisions/

# recent sessions
ls -t .claude/memory/sessions/ | head -5

# cross-tag (design AND a11y)
grep -rl "#design" .claude/memory/ | xargs grep -l "#a11y"
```

## Lifecycle

- **active** → current truth
- **deprecated** → no longer followed, kept for history
- **superseded** → replaced by another entry (pointed to by `supersedes`)

Never delete entries. Deprecate or supersede. Git history is the audit trail.

## Scope discipline

- `#template` entries propagate to client projects via `pnpm base:update`
- `#client-specific` entries are **never** written in the steaksoap template repo
- `scripts/base-patch.js` protects `.claude/memory/decisions|feedback|patterns|frictions|sessions` from overwrite on update
- **Client → template propagation** : when a client project writes a
  `#template` entry (decision / feedback / pattern / friction) that has
  general value, run `pnpm base:contribute` from the client repo. It
  detects the new `#template` artefact, clones steaksoap into
  `.upstream-cache/`, creates a `proposal/<client>-<date>-<topic>`
  branch, pushes it, and opens a PR via `gh`. The template maintainer
  reviews and merges (or declines). `sessions/*` are **never** propagated
  — extract the lesson into a feedback/pattern/friction first.

### File-level ownership inside `.claude/memory/`

| Path | Owner | On `base:update` |
|---|---|---|
| `TAGS.md` | **template** | propagates (overwrites client) |
| `README.md` | **template** | propagates (overwrites client) |
| `INDEX.md` | **client** (auto-generated) | protected |
| `decisions/` `feedback/` `patterns/` `frictions/` `sessions/` | **client** | protected |

If a client project needs a new tag, DON'T edit `TAGS.md` locally — your
change will be wiped on the next `pnpm base:update`. Instead: open an
issue/PR on the steaksoap template, land the tag there, then re-sync.
Tag vocabulary is a deliberate shared standard across all derived projects.

## What NOT to write to memory

- Code snippets (the code is the source of truth)
- File-path listings derivable from `ls` or `git ls-files`
- Transient todos (use TodoWrite)
- Secrets or credentials (ever)

## Why this protocol exists

Sessions lose context. Git history shows *what* but not *why chosen now over alternatives*. Memory fills the gap: persistent, taggable, greppable, versioned. Inspired by claude-mem without its runtime complexity.
