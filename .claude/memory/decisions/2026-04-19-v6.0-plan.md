# v6.0 Plan — Conductor & Workers (velocity refoundation)

Le propriétaire rapporte 3 douleurs quotidiennes :
1. Une petite modif prend 10 minutes (inner loop trop lent)
2. Le burn tokens augmente session après session
3. Zéro infrastructure pour travailler sur 4 clients en parallèle

## Diagnostic (après audit parallèle + docs Anthropic officielles)

La vraie cause **n'est pas** les bytes always-loaded (~23 KB baseline, gérable).
La vraie cause c'est :

1. **Tout tourne sur Opus 4.7** en main thread, y compris R0 (typos, rename,
   regen INDEX, memory wrap) → burn × 5 vs nécessaire.
2. **Aucune parallélisation** — 3 tâches demandées = 3 exécutions en série.
3. **`Agent` tool sous-utilisé** — délégué seulement pour recherche, jamais
   pour exécution.
4. **`pnpm validate` chaîne 11 steps systématiquement** même quand 3 suffisent.

## Décision : v6.0 = Conductor & Workers

Pattern hiérarchique : le main thread devient un **conductor** Opus qui
classifie le risque des tâches, alloue un worker du bon modèle, dispatche en
parallèle via `Agent` tool calls dans un seul message.

### Phases (exécution dans cet ordre)

1. **6.1 — Inner loop** (patch v5.8.2) : `pnpm validate:fast` = lint +
   typecheck + studio:typecheck en parallèle via npm-run-all. Heavy stuff
   (coverage, build, validators secondaires) reste dans `pnpm validate`
   (pre-PR / pre-release only).

2. **6.2 — Dispatcher flagship** (minor v6.0.0) :
   - 2 agents workers : `worker-haiku.md` (R0), `worker-sonnet.md` (R1)
   - Règle always-loaded `dispatch.md` avec taxonomie R0/R1/R2 + exemples
     steaksoap concrets (regen INDEX = R0, nouveau composant = R1, archi = R2)
   - Commande `/dispatch <tasks>` qui classifie, annonce, dispatche en
     parallèle, consolide les rapports, demande GO avant merge
   - Rapport de coût en fin de dispatch (tokens + wall-time par worker)
   - Mode proactif dans `critical.md` : même tâche unique R0 → déléguer
     à Haiku plutôt que main Opus

3. **6.3 — Memory refactor** (minor v6.1.0) :
   - `MEMORY.md` live digest (2-3 KB) = top 3 sessions + 5 decisions actives
     + frictions open, chargé en priorité sur INDEX.md
   - `decisions/releases/` sous-folder, exclus de l'INDEX principal
   - `sessions/YYYY-MM/` par mois, `sessions/archive/` > 30 jours avec
     résumé automatique (Haiku worker)
   - `pnpm memory:archive` standalone + trigger mensuel

4. **6.4 — Multi-client portfolio** (minor v6.2.0) :
   - `pnpm client:new <name>` → git worktree + client.md template + default model
   - `/morning-brief` slash command qui dispatche 4 workers Haiku (1 par
     worktree) en parallèle, produit tableau consolidé en ~15s
   - `client.md` gagne `model: sonnet|haiku|opus`

5. **6.5 — Context hygiene** (patch v6.2.1) :
   - Rules refactor : `principles.md` + `releases.md` + `i18n-sanity.md`
     passent en path-triggered (scopes ciblés)
   - `settings.json` : `autoCompactPercentage: 75`, `ENABLE_TOOL_SEARCH: auto`,
     permissions.allow étendu
   - `critical.md` section sur `/clear` entre tâches non-liées

## Taxonomie R0/R1/R2 (preview, formalisée en Phase 6.2)

| R0 — Haiku | R1 — Sonnet | R2 — Opus main |
|---|---|---|
| rename variable/file | nouveau composant UI | décision archi |
| regen INDEX / memory wrap | nouvelle page | debug sans repro clair |
| bump version string | nouveau champ schema | perf non-trivial |
| ajouter clé locale | bug fix avec repro | refactor multi-layer |
| update CHANGELOG | ajouter une commande slash | design direction |
| file move | nouveau hook suivant pattern | revue sécurité |

## Impact attendu cumulé

| Symptôme | Avant | Après v6.0-v6.2 |
|---|---|---|
| Petite modif | 10 min Opus | ~30 s Haiku worker |
| `pnpm validate` ad-hoc | ~30 s | ~6-8 s (`validate:fast`) |
| Session longue | dégradation à 95% | stable jusqu'à 75% |
| Travail 4 clients | séquentiel | parallèle réel |
| Coût tokens/session | 100% Opus | ~40% Opus + 60% Sonnet/Haiku |
| INDEX à 6 mois | 300+ entries | < 50 KB stable |

## Ce qu'on NE fait PAS

- Prompt caching manuel (Anthropic le gère auto sur Claude Code)
- Fusionner sizing.md + components.md (concerns différents)
- Worktree isolation par sous-agent (overhead > gain in-process)
- Supprimer des slash commands sans évidence d'orphelin
- Toucher aux 13 leçons HDVA / schemas Sanity / validators content stack

## Rationale "le propriétaire m'a délégué le choix"

Le propriétaire a explicitement dit : *"c'est TOI qui va décider, je n'ai
pas la compréhension ni le recul. Connais mon objectif, fais le meilleur
plan."* Objectif = vitesse + productivité + 4 clients + propreté code. Ce
plan priorise le ROI daily-friction (6.1 → 6.2) sur l'esthétique
structurelle (6.3 → 6.5).
