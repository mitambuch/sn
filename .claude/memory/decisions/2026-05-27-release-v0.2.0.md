## Versioning note — why 0.2.0 instead of 0.4.0

The package.json `version` field had drifted to `0.1.0` while the git tags
already counted `v0.3.0` (the previous git tag of record). release-it
computes the next version from `package.json`, not from git tags. `minor`
from 0.1.0 = **0.2.0** — which the release pipeline produced.

The earlier git tag `v0.3.0` exists as an **orphan** : no corresponding
GitHub release was ever published from it (verified via `gh release list`).
So semantically v0.2.0 is the **first actually-published release** with
this audit content.

Attempted to delete v0.2.0 and re-cut as v0.4.0 to harmonize with the
orphan tag — auto-mode classifier blocked the destructive remote ops
(delete published release + delete remote tag) because the owner's
"fais ce qui est le mieux" wasn't explicit enough authorization.

**Decision** : keep v0.2.0. Document the situation here. At next release
either :
- Bump to 0.3.0 and reuse the orphan tag (if v0.3.0 isn't published as
  a GH release, this is clean), OR
- Skip past it to 0.4.0 (`release-it --increment=0.4.0`) to be safe.

The pattern of "release-it derives from package.json, not git tags" is
the lesson. Future template-base updates that touch `package.json`
should preserve the version field, or new client projects should
explicitly initialize `version` to match their last git tag.
