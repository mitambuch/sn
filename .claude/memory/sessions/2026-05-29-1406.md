## Frictions rencontrées

### 1. Nouvelles API keys Supabase n'ont pas les grants tables

Owner a fourni un `sb_secret_*` (nouveau format API keys Supabase
introduit en 2024). Il fonctionne pour `auth.admin.*` (création
user OK) mais retourne `permission denied for table profiles /
invitation_codes / share_codes / inquiries / access_requests` sur
toute opération REST. Cause probable : la migration 0010
security_hardening a fait des GRANTs column-level pour
`authenticated` sans miroir pour le nouveau role mappé par
`sb_secret_*`. Le legacy service_role JWT (eyJ...) aurait sans
doute marché.

**Workaround appliqué** : script SQL à exécuter en role `postgres`
via le SQL Editor (qui bypasse column GRANT + RLS). 30 secondes
owner.

**Reste à faire (v1.2.x)** : ajouter une migration qui étend les
grants opérateur au nouveau role `service_role` mappé par
`sb_secret_*`, ou documenter explicitement l'usage du legacy JWT
pour les ops scripts. Voir `studio/scripts/push-site-config.js`
(qui marche en `sb_secret_*` pour Sanity — différent backend) vs
`scripts/create-admin-user.js` (qui marche partiellement).

### 2. Studio Sanity pas déployé

Pas d'URL stable pour Salva. `pnpm studio:deploy` nécessite
interactivité (login Sanity CLI + choix du hostname slug). Pas
autoable sans risque (slug actuel `steaksoap-studio` est le nom
template, pas Sawnext-spécifique). À traiter en session dédiée
quand owner peut taper le hostname.

### 3. Footer Sanity invisible sur homepage

Découvert pendant le post-mortem : la home utilise `LandingFooter`
(features/landing/), pas le `Footer` public que j'ai câblé.
Le wire-up siteConfig n'affiche les valeurs Sanity que sur les
pages dans `PublicLayout` (`/login`, `/share/:code` et futur
`/about`, `/contact`). À régler en v1.2.x avec un mirror du
wire sur LandingFooter, ou en assumant que la home a sa propre
voix.
