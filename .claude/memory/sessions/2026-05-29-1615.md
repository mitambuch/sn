# Session — v1.2.1 patch : login submit blanc

## Goal

15:46 — owner repère que le bouton submit de `/login` n'est plus
blanc (régression visuelle pendant qu'il montre au client).
Diagnostic + fix surgical.

## Diagnostic

Le submit utilisait `border-border bg-fg text-bg` dans
`<section data-theme="dark">`. Sur le papier la cascade fait que
`bg-fg` = `#f0f0f0` (blanc) dans la zone data-theme=dark.

En pratique sur le live deploy, la cascade `[data-theme='dark']`
ne gagnait pas dans le contexte de cette section overlay
(html-level theme overrides, ou compute issue Tailwind 4 avec les
variables) — le bouton rendait sombre.

Le bouton "Demander un accès" du landing (Hero) utilise lui
`bg-white text-black` (literals), garantis blancs en toute
circonstance. Asymétrie de robustesse.

dans les 2 modes** (light et dark) per `src/index.css`. Garantie de
blanc quel que soit le contexte cascade. Bordure retirée (cohérence
avec le CTA primary du landing).

## Pattern à retenir (à documenter en pattern memory v1.3.x)

Pour les **buttons garantis blancs/noirs indépendamment du theme** :
préférer les tokens `on-ink` / `ink` aux tokens `fg` / `bg`.
- `bg-fg text-bg` → dépend du data-theme cascade, fragile en overlay
- `bg-on-ink text-ink` → toujours blanc/noir, robuste

Règle générale : utiliser `fg`/`bg` quand le composant doit
RESPECTER le theme courant ; utiliser `on-ink`/`ink` quand il doit
FORCER une lecture sombre-fond ou claire-fond.
