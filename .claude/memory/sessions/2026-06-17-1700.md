# Session — Événements publics sur la home (vrais + fiches floutées)

Owner : le client a rempli Sanity et tagué de vrais events `visibility: public`.
Veut les voir sur la home publique, en gardant quelques fiches floutées pour le FOMO.

## Clarification clé (anti-souvenir)

L'« espace catalogue flouté » dont l'owner se souvenait n'existait **pas** sur la
home publique — c'était le catalogue **membre** caché par `FEATURES.catalogueLive:
false`. La home n'avait aucune section events. J'ai donc **créé** une section, pas
dé-flouté quelque chose. Surface confirmée via AskUserQuestion : home publique + 3 floutées.

## Fait (branche `feat/landing-public-events`, commit `fd989d1`, non mergé)

- **Section S06 `LandingEvents`** entre Domains (S05) et Access (S08). Vrais events
  publics (Sanity) + 3 fiches **frosted/locked** ; tout clic (vraie OU floutée) ouvre
  la demande d'accès. Funnel, pas catalogue.
- **`usePublicEvents`** gate-aware : action publique `publicEvents` de la fonction
  Netlify quand le gate audience est ON, lecture Sanity directe sinon. **Zéro mock
  fallback** — la home publique ne montre jamais de fausses fiches → section absente
  tant qu'aucun event public réel n'existe.
- **`GROQ_PUBLIC_EVENTS`** (strict `visibility == "public"`, projection lean, cap 8)
  + type `PublicEvent` + action serveur dans `catalogue.mts`.
- Mécaniques : `magnetic-hover` (vraies cartes) + `masked-reveal` (voile flouté qui se
  lève partiellement au hover). Réutilise Card / MagneticHover / Reveal / SectionTag.
- i18n `landing.events.*` + index `s06*` (FR/EN/ES, parité), footerCount → 08.
- Test `LandingEvents.test.tsx` (4 cas) vert.
