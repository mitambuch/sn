# Session — Page salon /QR (présentation + QR scannable + lead form)

Suite de la session CMS/release de la veille. Objectif owner : un **QR fixe**
pour un salon — le client ouvre `saw-next.ch/QR`, montre un QR à l'écran, les
gens scannent et atterrissent sur une page de présentation quali, mobile-first,
multilingue.

## Livré (mergé main `e92dce7`, déployé)

Page **/QR** (route top-level hors LocaleProvider), construite en 5 phases +
overlay QR, chaque phase un commit :

1. Route + shell (chrome BrandMark + switcher, hero à mot cyclant).
2. Sections : 10 domaines (réutilise `landing.domains.*`), 4 piliers qualité,
   espace membre (3 points) — masked-reveal via le composant `Reveal` existant.
3. Formulaire sur-mesure → `access_requests` (canal anonyme → email Salva),
   marqué `activity: "Salon / QR"`. **submitInquiry ne marche pas en anon**
   (il simule sans email) → d'où access_requests.
4. CTA bas : Devenir membre (scroll vers form), Voir le site, **Partager**
   (native-web-share + fallback clipboard).
5. Tests (5) + validate complet.
6. **Overlay QR** : bouton « QR » header → QR plein écran (qrcode.react, SVG)
   encodant `https://saw-next.ch/QR`, fond blanc/modules noirs. C'est LE
   mécanisme salon.

FR/EN/ES (parité), `noindex` (exclusivité salon), zéro fetch Sanity. Pattern
extrait : [[standalone-public-page-outside-localeprovider]].
